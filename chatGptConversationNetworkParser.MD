# chatGptConversationNetworkParser.ts 정밀 테스트 시나리오 (50개)

## [그룹 1: 데이터 포맷 및 소스 감지] (1-10)
1. **표준 JSON**: `mapping`과 `current_node`가 포함된 표준 객체 파싱 성공 여부.
2. **RSC 디코딩**: `streamController.enqueue`로 감싸진 유니코드 이스케이프 문자열의 정상 복원.
3. **SSE 스트림 조합**: 5개 이상의 `delta` 조각이 들어왔을 때 순서대로 텍스트가 합쳐지는가.
4. **HTML 임베디드**: `<script>` 태그 내부에 숨겨진 JSON 데이터를 찾아낼 수 있는가.
5. **Next.js Hydration**: `__next_f.push` 패턴 내의 RSC 페이로드 추출 성공 여부.
6. **복합 데이터**: 한 레코드에 JSON과 HTML이 섞여 있을 때 점수가 높은 쪽을 우선하는가.
7. **BOM/접두어 처리**: JSON 앞에 `for(;;);`와 같은 보안 접두어가 붙어있어도 제거하고 파싱하는가.
8. **비정상 JSON**: 문법이 틀린 JSON 수신 시 앱이 크래시되지 않고 Fallback(HTML)으로 넘어가는가.
9. **대용량 페이로드**: 5MB 이상의 거대 대화 데이터를 파싱할 때 성능 저하나 메모리 누수가 없는가.
10. **MIME 타입 무시**: Content-Type이 text/plain이어도 내용이 JSON이면 파서가 작동하는가.

## [그룹 2: 메시지 체인 복원 로직] (11-20)
11. **선형 경로 추적**: `current_node`부터 `parent`를 따라 루트까지 완벽한 단일 체인을 형성하는가.
12. **분기(Branch) 처리**: 여러 대화 분기가 있을 때 '현재 선택된 경로'의 메시지만 정확히 가져오는가.
13. **루트 노드 탐색**: 부모가 없는 최상위 노드(`root`)를 정확히 식별하여 시작점으로 삼는가.
14. **순서 정렬**: 생성 시간(`create_time`)과 부모 관계가 일치하지 않을 때 부모 관계를 우선하여 정렬하는가.
15. **중간 노드 누락**: 중간에 부모 ID가 없는 노드가 발견될 때 가능한 최선의 연결을 시도하는가.
16. **역할 매핑 (System)**: `role: system` 메시지를 내부적으로 필터링하거나 표시 여부를 결정하는가.
17. **역할 매핑 (Tool)**: `role: tool` 메시지 중 이미지가 없는 단순 로그성 메시지를 제외하는가.
18. **역할 재매핑**: `role: tool`이지만 이미지 페이로드가 있다면 `assistant`로 변환하여 보여주는가.
19. **사용자 이름**: `author.name`이 있을 경우 `role`보다 우선하여 표시 이름을 추출하는가.
20. **빈 메시지**: 내용이 없는 노드가 체인 중간에 있을 때 무시하고 다음 노드로 연결하는가.

## [그룹 3: 필터링 및 가시성] (21-30)
21. **내부 채널 필터**: `channel: commentary`인 메시지가 사용자에게 노출되지 않는가.
22. **비공개 수신자**: `recipient`가 `all`이 아닌 특정 플러그인 전용인 메시지 제외 여부.
23. **Hidden 플래그**: `metadata.is_visually_hidden_from_conversation` 속성 존중 여부.
24. **삭제된 메시지**: `metadata.is_redacted: true`인 메시지 처리 로직 검증.
25. **상태값 검증**: `status`가 `finished_successfully`가 아닌(진행 중인) 메시지 처리 방식.
26. **Deep Research 필터**: 중간 과정인 `thoughts` 노드를 최종 결과에서 제외하는가.
27. **Reasoning Status**: `reasoning_status: is_reasoning`인 노드 제외 여부.
28. **중복 메시지**: 내용과 ID가 완전히 동일한 노드가 중복 수신될 때 데드랍 처리.
29. **요약(Summary) 생성**: 첫 번째 메시지를 기반으로 80자 이내의 요약이 정상 생성되는가.
30. **제목 정규화**: `ChatGPT - `와 같은 접두사가 제목에서 깔끔하게 제거되는가.

## [그룹 4: 메타데이터 및 출처(Citation)] (31-40)
31. **인라인 토큰 치환**: `\ue200cite\ue202...` 토큰이 `【n】` 또는 텍스트로 치환되는가.
32. **출처 배열 구축**: `content_references`를 분석하여 `sources` 배열을 순서대로 생성하는가.
33. **URL 중복 제거**: 서로 다른 노드에서 동일한 URL을 참조할 때 `sources` 배열에서 하나만 남기는가.
34. **괄호 제거 로직**: 파서가 강제로 넣은 `(【1】)` 형태의 괄호를 본문에서 제거하는가.
35. **검색 결과 매핑**: `search_result_groups`의 정보를 `sources`로 통합하는가.
36. **대체 텍스트(Alt)**: 출처 링크의 `alt` 값이 없을 때 사이트 도메인을 제목으로 사용하는가.
37. **스니펫(Snippet)**: 검색 결과의 요약 문구가 `description` 필드에 정확히 들어가는가.
38. **출처 인덱스 정합성**: 본문의 `【1】` 숫자가 `sources[0]`과 정확히 일치하는가.
39. **잘못된 토큰**: 닫는 유니코드(`\ue201`)가 없는 깨진 토큰 수신 시 텍스트 파싱 중단 방지.
40. **파일 참조**: `filecite` 타입의 토큰을 `[파일 참조]`와 같은 읽기 쉬운 문구로 바꾸는가.

48. **Widget State**: `WIDGET_STATE_MARKER`를 포함한 대형 객체에서 유효 대화만 추출하는가.
49. **Deep Research 리포트**: 복수의 리포트 메시지가 있을 때 이를 하나의 메시지로 합치거나 순서대로 나열하는가.
50. **Fallback URL**: 유효한 URL이 없을 때 `fallbackUrl`이 `sourceUrl`로 정확히 할당되는가.

## [그룹 6: 에러 핸들링 및 예외 상황] (51-60)
51. **빈 페이로드**: 네트워크 응답 본문이 완전히 비어있을 때 앱이 멈추지 않고 null을 반환하는가.
52. **Malformed RSC**: 디코딩은 성공했으나 내부 배열 구조가 깨진 RSC 데이터 처리.
53. **타임아웃 시뮬레이션**: 스트리밍 중 `[DONE]` 신호 없이 연결이 끊겼을 때 지금까지의 데이터를 정상 복원하는가.
54. **순환 참조 JSON**: 악의적이거나 비정상적인 순환 참조가 포함된 JSON을 파싱할 때 무한 루프를 방지하는가.
55. **중첩 한도 초과**: `extractWidgetStatePayload` 등에서 재귀 깊이(depth) 한도를 초과할 때 스택 오버플로우 방지.
56. **알 수 없는 역할**: `author.role` 필드에 'assistant', 'user', 'tool' 외의 미상 문자열이 들어올 경우의 방어 로직.
57. **이상한 인코딩**: UTF-8이 아닌 깨진 문자열이 포함되었을 때 파서가 에러를 뱉지 않고 건너뛰는가.
58. **비정상 시간값**: `create_time`에 문자열이나 NaN이 들어올 경우 정렬 로직이 터지지 않고 fallback 값을 사용하는가.
59. **잘못된 트리 구조**: 부모가 자신을 가리키는 등 트리 루프가 발생했을 때 탐색이 중단되는가.
60. **Missing Mapping Key**: RSC 배열에서 `mapping`이나 `current_node` 키가 누락되었을 때의 안전한 실패 처리.

## [그룹 7: 동시성 및 복합 스트림 처리] (61-70)
61. **다중 스트림 겹침**: 여러 개의 SSE 스트림이 동시에 들어올 때 식별자(ID) 기준으로 정확히 분리하여 조립하는가.
62. **순서 뒤섞임**: 네트워크 지연으로 SSE 패치가 순서가 섞여 들어왔을 때 최종 상태가 일관되게 유지되는가.
63. **중간 삭제 패치**: 스트리밍 도중 특정 파트를 지우는(`replace` with empty) 패치에 정상 대응하는가.
64. **상태값 롤백**: `status`가 `finished`에서 다시 `in_progress`로 번복될 경우의 처리.
65. **동시성 경쟁 상태**: HTML 파싱 중 JSON 파싱 이벤트가 발생했을 때 파서 간의 충돌 여부.
66. **대용량 청크 분할**: 한 번의 SSE 이벤트 텍스트가 너무 길어서 여러 청크로 쪼개졌을 때 재결합 기능.
67. **오래된 노드 덮어쓰기**: 동일 노드에 대해 과거 타임스탬프의 패치가 뒤늦게 도착했을 때 무시하는가.
68. **병렬 위젯 상태**: 하나의 응답에 여러 개의 서로 다른 위젯 상태가 갱신될 때 최신 상태를 병합하는가.
69. **Patch 인덱스 누락**: 배열 패치 중 중간 인덱스가 건너뛰어질 때 빈 요소를 안전하게 다루는가.
70. **메시지 파편화**: 텍스트, 이미지, 출처 정보가 각각 다른 이벤트로 아주 잘게 쪼개져 올 때의 완전한 조립.

## [그룹 8: 파싱 성능 및 최적화] (71-80)
71. **정규식 백트래킹(ReDoS)**: 극단적으로 길고 복잡한 마크다운 코드에 정규식을 적용할 때 성능이 저하되지 않는가.
72. **가비지 컬렉션**: 대규모 매핑 객체를 파싱한 후 불필요한 참조가 즉시 해제되는가 (WeakSet 활용 확인).
73. **대량 노드 정렬**: 노드가 10,000개 이상일 때 트리를 선형으로 복원하는 시간 복잡도 검증.
74. **문자열 할당 최적화**: 수많은 `split`과 `join` 연산 시 불필요한 메모리 할당이 최소화되는가.
75. **조기 종료(Early Exit)**: 유효하지 않은 데이터로 판명된 즉시 파싱을 중단하여 CPU 리소스를 절약하는가.
76. **캐싱 히트율**: `resolvedIndexCache` 등 내부 캐시의 작동 효율성 검증.
77. **HTML 탐색 효율**: 수 MB의 HTML 파일에서 정규식 탐색 시 브라우저 스레드 블로킹 방지.
78. **객체 순회 속도**: `Object.values()` 재귀 호출이 깊은 중첩 구조에서 지연을 유발하지 않는가.
79. **스코어링 연산**: `scoreConversation` 계산 시 배열 순회가 최적화되어 있는가.
80. **중복 검사 성능**: `seenFingerprints.has`의 해시 탐색 성능 검증.

## [그룹 9: 특수 마크다운 및 확장 문법] (81-90)
81. **수식 이스케이프**: LaTeX 문법 내의 백슬래시(`\`)나 괄호가 파싱 중 깨지지 않고 보존되는가.
82. **중첩 코드 블록**: 코드 블록 안에 또 다른 코드 블록 문법(백틱)이 있을 때 오작동 방지.
83. **다이어그램 텍스트**: Mermaid 외에 PlantUML 등 다른 다이어그램 텍스트의 코드 블록 래핑.
84. **인라인 HTML**: 사용자 또는 AI가 작성한 `<script>`, `<iframe>` 등이 이스케이프되거나 필터링되는가.
85. **커스텀 플러그인 응답**: 특정 플러그인만이 반환하는 독자적인 JSON 메타데이터 구조 파싱.
86. **보이지 않는 유니코드**: 제로 위스 스페이스(ZWSP)나 LTR/RTL 마크가 포함된 텍스트의 정규화.
87. **다국어 혼합**: CJK(한중일), 아랍어, 이모지가 복잡하게 섞인 텍스트 스트림의 올바른 분리 및 결합.
88. **리스트 내 들여쓰기**: 들여쓰기된 깊은 리스트 구조가 텍스트 정규화 과정에서 뭉개지지 않는가.
89. **표(Table) 파싱**: GFM 테이블 문법 앞뒤의 줄바꿈 규칙이 정확히 적용되는가.
90. **특수 기호 출처**: 출처 이름이나 URL에 `)`, `]`, `"` 등 파싱을 헷갈리게 하는 특수문자가 포함된 경우.

## [그룹 10: 다양한 플랫폼/브라우저 환경 대응] (91-100)
91. **Line Ending 차이**: CRLF(`\r\n`)와 LF(`\n`) 혼재 시 텍스트 정규화가 일관되게 작동하는가.
92. **레거시 포맷**: ChatGPT 초창기의 오래된 데이터 포맷(현재와 스키마가 약간 다른 JSON) 호환성.
93. **URL 스키마 차이**: `http://`가 생략된 도메인이나 `//`로 시작하는 프로토콜 상대 URL 정규화.
94. **브라우저 확장 개입**: 타사 브라우저 확장 프로그램이 DOM에 삽입한 불필요한 스크립트 무시 여부.
95. **네트워크 에러 코드**: 403, 429, 500 등 에러 상태 코드 응답 시 파서가 데이터 없음을 정상 처리하는가.
96. **빈 응답 본문**: 200 OK이나 본문 크기가 0바이트인 경우.
97. **모바일 뷰 응답**: 모바일 User-Agent로 요청되어 약간 다르게 렌더링된 HTML의 파싱.
98. **웹뷰/IFrame 내부**: IFrame 내부에서 캡처된 네트워크 로그의 상대 경로 해결.
99. **대용량 헤더**: 본문보다 HTTP 헤더가 비정상적으로 큰 레코드의 처리.
100. **미래 스키마 대비**: `message.content`에 텍스트나 배열이 아닌 새로운 객체 타입이 추가되었을 때 에러 없이 무시/우회하는가.

## [그룹 11: 딥 리서치 및 복잡한 추론 (Deep Research & Reasoning)] (101-110)
101. **초기 추론 플래그**: `metadata.reasoning_status`가 `is_reasoning`에서 `reasoning_ended`로 변경될 때 정상 반영되는가.
102. **생각 블록 병합**: `thoughts` 배열 내부에 여러 개의 추론 단계가 있을 때 순서대로 조합하여 하나의 블록으로 복원하는가.
103. **스테어(Steer) 페이로드**: `isDeepResearchSteerPayloadText` 함수가 `connector_openai_deep_research`를 정확히 감지하는가.
104. **리서치 타임아웃**: 리서치 도중 실패(Timeout) 메시지가 내려왔을 때 진행 상황까지의 데이터라도 복원하는가.
105. **다중 리포트 생성**: 하나의 대화 노드에 여러 개의 `report_message`가 달렸을 때 시간순으로 모두 수집되는가.
106. **위젯 상태와 본문 분리**: `widget_state`가 포함된 노드에서 본문 텍스트와 위젯 데이터를 섞이지 않게 분리하는가.
107. **커스텀 플러그인 혼합**: Deep Research와 일반 웹 검색 플러그인이 동시에 호출되었을 때 출처가 병합되는가.
108. **추론 전용 토큰**: 추론 과정에서만 나타나는 특수 토큰(`\ue200` 계열 외)을 본문에서 숨기는가.
109. **중첩된 리서치 컨텍스트**: 리서치가 다른 리서치를 하위 태스크로 불렀을 때 트리 뎁스(depth)를 잃지 않는가.
110. **리서치 진행률 표시**: `content_type: reasoning_recap`을 파싱하여 소요 시간(초)을 정상 추출하는가.

## [그룹 12: 에이전트 및 툴 호출 (Agent & Tool Calls)] (111-120)
111. **툴 호출 시작 감지**: `author.role === 'tool'`이 시작될 때 이를 시각적으로 숨김 처리(기본값)하는가.
112. **파이썬 코드 실행 결과**: `tool_calls` 내 파이썬 샌드박스 실행 결과(stdout/stderr)가 텍스트로 잘 파싱되는가.
113. **샌드박스 에러 처리**: 샌드박스 실행 중 발생한 예외(Exception) 로그가 파서 오류를 유발하지 않는가.
114. **파일 다운로드 툴**: 툴 응답에 `backend-api/files/download/` 형태의 URL이 포함 시 이를 파일 링크로 변환하는가.
115. **도형/차트 생성**: 툴이 생성한 SVG나 Mermaid 코드가 일반 텍스트가 아닌 다이어그램 블록으로 식별되는가.
116. **DALL-E 3 이미지**: DALL-E 3 툴이 반환한 `image_asset_pointer`가 올바른 이미지 마크다운으로 치환되는가.
117. **다중 툴 병렬 호출**: 한 번에 여러 개의 툴(예: 검색 3번 동시 실행)이 호출된 결과를 하나의 노드에 잘 합치는가.
118. **보이지 않는 툴 응답**: `metadata.is_visually_hidden_from_conversation`이 툴 응답에 적용되었을 때 제외 여부.
119. **사용자 중단 툴**: 툴 실행 중 사용자가 Stop을 눌렀을 때 `status: aborted` 처리가 파서를 멈추지 않는가.
120. **커스텀 Action**: GPTs나 커스텀 Action이 반환한 예상치 못한 구조의 JSON을 무시하고 텍스트만 건지는가.

## [그룹 13: 이미지 및 파일 첨부 엣지 케이스 (Image & File Attachments)] (121-130)
121. **유효하지 않은 Base64**: 손상된 Base64 인코딩 문자열이 `IMAGE_URL_PATTERN`에 의해 걸러지거나 예외 처리되는가.
122. **지원하지 않는 포맷**: `.tiff`나 `.heic` 등 웹 미지원 포맷 링크 수신 시 일반 파일 링크로 대체하는가.
123. **Sediment 포인터 누락**: `sediment://` 뒤에 ID가 없는 비정상 패턴을 무시하는가.
124. **파일 서비스 포인터**: `file-service://` 패턴이 `[이미지 첨부: ...]` 텍스트로 정확히 치환되는가.
125. **멀티모달 프롬프트**: 사용자가 입력한 여러 장의 이미지가 `parts` 배열에서 순서대로 추출되는가.
126. **워터마크 이미지**: `watermarked_asset_pointer` 키를 가진 이미지를 우선하거나 동일하게 취급하는가.
127. **대용량 메타데이터 이미지**: 이미지 메타데이터 자체가 1MB가 넘을 때 `collectImageParts` 함수가 지연되지 않는가.
128. **중복 이미지 필터링**: 동일한 이미지 포인터가 `content`와 `metadata` 양쪽에 있을 때 한 번만 표시하는가.
129. **손상된 파일 객체**: `file_type`은 명시되었으나 URL이나 데이터가 없는 객체에 대한 안전한 스킵.
130. **오디오/비디오 첨부**: 이미지 이외의 미디어(`audio/mp3` 등)를 식별하고 파싱 에러를 내지 않는가.

## [그룹 14: RSC 특이 패턴 및 React Node (RSC Specifics)] (131-140)
131. **Promise 노드**: RSC 페이로드 내에서 아직 Resolve되지 않은 Promise 노드를 안전하게 무시하는가.
132. **Client Reference**: 클라이언트 컴포넌트 참조(`$L1` 등)가 섞여 있어도 파서가 다운되지 않는가.
133. **거대 참조 딕셔너리**: `_1`, `_2` 등 1,000개가 넘는 RSC 레퍼런스를 해소할 때 `resolveReference` 캐시가 효과적인가.
134. **재귀적 RSC 배열**: RSC 배열 내에 또 다른 RSC 트리가 중첩되어 있을 때 `flat`하게 잘 풀어내는가.
135. **HTML 혼합 RSC**: RSC 문자열 조각 안에 완전한 HTML이 문자열로 들어있을 때 이스케이프가 깨지지 않는가.
136. **분할된 streamController**: `streamController.enqueue`가 수십 개로 나뉘어져 있을 때 가장 큰 페이로드를 정확히 찾는가.
137. **Index Out of Bound**: RSC 참조 인덱스가 배열 크기를 초과할 때 null을 반환하며 죽지 않는가.
138. **Map / Set 객체**: RSC 디코딩 시 Map이나 Set 타입이 들어왔을 때 Object/Array로 대체 처리되는가.
139. **Symbol 노드**: React Symbol(예: `Symbol(react.element)`)을 만났을 때 일반 객체 탐색에서 제외하는가.
140. **RSC 내 빈 문자열**: 내용이 없는 빈 텍스트 노드가 대화 배열에 쓰레기 데이터로 남지 않는가.

## [그룹 15: SSE 델타 인코딩 한계 상황 (SSE Edge Cases)] (141-150)
141. **단일 바이트 패치**: 글자가 1바이트씩(`event: delta` 수백 번) 날아올 때 `currentParts` 업데이트 성능.
142. **이모지 쪼개짐**: 4바이트 이모지(`👩‍🚀`)가 두 개의 패치로 쪼개져 올 때 깨짐 없이 결합되는가.
143. **패치 인덱스 오류**: `patch.p` 경로가 `/message/content/parts/99`처럼 범위를 벗어날 때의 처리.
144. **잘못된 o(Operation)**: 패치의 `o` 값이 `append`나 `replace`가 아닌 `remove`, `copy` 등일 때 무시하는가.
145. **누락된 v(Value)**: 패치에 `v` 값이 없을 때 `undefined` 문자열이 본문에 붙지 않는가.
146. **Add 이벤트 중복**: 동일한 `message_id`로 `add` 이벤트가 두 번 들어올 때 기존 데이터를 덮어쓰거나 무시하는가.
147. **빈 델타 데이터**: `data: {}` 처럼 내용이 없는 이벤트 수신 시 루프를 안전하게 다음으로 넘기는가.
148. **스트림 재연결**: 네트워크 불안정으로 동일 대화에 대한 SSE 스트림이 다시 처음부터 쏟아질 때의 처리.
149. **메시지 순서 역전**: `parent_id` 기반 트리가 완전히 꼬인 스트림(과거 메시지가 나중에 옴) 수신 시 정렬.
150. **데이터 없는 [DONE]**: 아무 데이터 없이 `[DONE]`만 들어왔을 때 파서가 `null`을 반환하는가.

## [그룹 16: 중첩된 Conversation Mapping (Nested Mappings)] (151-160)
151. **트리 깊이 제한**: 메시지 트리 깊이가 100 이상 깊어질 때 `getOrderedMappingNodes`가 Call Stack Exceeded를 피하는가.
152. **형제 노드 병합**: 하나의 부모에 여러 자식이 있을 때 `current_node`가 가리키는 가지(Branch)만 정확히 취하는가.
153. **고아 노드(Orphan)**: 부모도 없고 자식도 없는 붕 뜬 노드가 매핑에 있을 때 안전하게 무시하는가.
154. **루트 노드 다수**: `parent_id`가 null인 노드가 여러 개일 때 가장 최신의 `create_time`을 루트로 잡는가.
155. **Circular Parent**: 노드 A의 부모가 B이고, B의 부모가 A인 순환 참조 시 무한 루프에 빠지지 않는가.
156. **동일 생성 시간**: 여러 노드의 `create_time`이 완전히 동일할 때 노드 ID 알파벳 순 등 결정적(deterministic) 정렬이 되는가.
157. **빈 매핑 객체**: `mapping: {}`인 상태로 파서에 진입했을 때 `null` 또는 빈 배열을 정상 반환하는가.
158. **삭제된 브랜치**: 사용자가 재생성하여 버려진(abandoned) 브랜치의 데이터가 결과에 섞이지 않는가.
159. **루트 변경**: 시스템 프롬프트 변경 등으로 루트 노드의 ID가 도중에 바뀌었을 때의 트래킹.
160. **디버그 노드**: `metadata.is_debug: true`와 같은 내부 테스트용 노드를 프로덕션에서 걸러내는가.

## [그룹 17: 악의적 페이로드 및 보안 (Security & Malformed)] (161-170)
161. **XSS 페이로드**: 메시지 본문에 `<script>alert(1)</script>`가 포함 시 그대로 문자열로 렌더링되게 이스케이프하는가.
162. **초거대 문자열**: 10MB 크기의 단일 텍스트 문자열이 들어올 때 정규화 함수(`normalizeMessageText`)가 멈추지 않는가.
163. **프로토타입 오염**: JSON 페이로드 키에 `__proto__`나 `constructor`가 있을 때 파서가 오염되지 않는가.
164. **무한 배열**: `content.parts`에 수만 개의 빈 문자열이 들어있을 때 렌더링 전 가지치기(trim) 하는가.
165. **악의적 URL**: 출처 URL에 `javascript:void(0)`나 `vbscript:`가 포함되어 있을 때 스크러빙하는가.
166. **이중 인코딩**: 이미 한 번 디코딩된 유니코드 텍스트를 또 디코딩하여 깨뜨리지 않는가.
167. **거짓 마커**: 텍스트 본문에 `The latest state of the widget is:`라는 글자를 사용자가 고의로 입력 시 위젯 파서가 오작동하지 않는가.
168. **가짜 출처 토큰**: 사용자가 `\ue200cite...` 토큰을 직접 채팅에 쳤을 때 치환 로직이 에러를 일으키지 않는가.
169. **비밀번호/키 포함**: 메시지에 API 키 등이 있을 때, 이를 특별히 검열하지 않고 원본 그대로 파싱하는가(검열은 렌더러/사용자 몫).
170. **SQL 인젝션 패턴**: 페이로드 내 SQL 구문과 유사한 패턴이 있을 때 단순 텍스트로 안전하게 파싱하는가.

## [그룹 18: 메모리 및 성능 한계 (Performance Limits)] (171-180)
171. **정규식 캐싱**: 여러 번 호출되는 패턴 매칭 함수에서 정규식이 매번 새로 생성되지 않게 상수화되어 있는가.
172. **메시지 1000개 파싱**: 대화 내 메시지가 1000개가 넘을 때 파싱 소요 시간이 1초(1000ms) 이내인가.
173. **대량의 출처(Sources)**: 한 메시지에 출처가 50개 이상 달려있을 때 dedup(중복 제거) 로직의 효율성.
174. **연쇄 replace 최적화**: `normalizeMessageText`에서 체인 형태의 `replace`가 문자열 재할당을 과도하게 일으키지 않는가.
175. **배열 평탄화(flatMap)**: 깊게 중첩된 배열을 펼칠 때 콜스택 오버플로우가 나지 않는가.
176. **스코어링 부하**: `scoreConversation` 계산 시 매우 긴 텍스트의 `length`와 정규식 매치가 병목을 일으키지 않는가.
177. **Map 조회 성능**: `orderedNodeIdToIndex.has()` 등 해시 맵 조회가 O(1)로 처리되고 있는가.
178. **Garbage 생성 최소화**: 파싱 중 버려지는 임시 객체 생성을 최소화하여 GC(가비지 컬렉션) 멈춤(Pause)을 막는가.
179. **Worker 스레드 이전**: 만약 파싱이 너무 오래 걸릴 경우 이를 Web Worker로 넘기기 좋은 순수 함수 형태를 유지하는가.
180. **Set 중복 제거**: 배열의 중복 요소를 제거할 때 `Array.from(new Set())`의 성능 한계 확인.

## [그룹 19: 프롬프트 및 사용자 컨텍스트 (Prompt & Context)] (181-190)
181. **유저 프롬프트 유지**: 사용자가 입력한 엔터(줄바꿈)나 탭(들여쓰기) 형식이 파싱 후에도 보존되는가.
182. **에디터 컨텍스트**: Cursor나 IDE에서 보낸 `[File: src/main.ts]` 형태의 컨텍스트 텍스트가 정상 분리 파싱되는가.
183. **유저 첨부 이미지**: 사용자가 업로드한 이미지가 텍스트 앞에 `![image](...)` 형태로 올바르게 전처리되는가.
184. **멀티라인 질문**: 사용자의 질문이 아주 긴 여러 문단일 때 잘림 없이 끝까지 파싱되는가.
185. **명령어 패턴**: `/help`, `/search` 등 슬래시 명령어가 포함된 사용자 메시지 파싱.
186. **커스텀 인스트럭션**: `metadata`에 숨겨진 커스텀 인스트럭션이 본문 텍스트에 노출되지 않는가.
187. **잘린 프롬프트 복원**: 네트워크 전송 중 잘린 사용자 프롬프트 조각이 있을 경우 대응.
188. **빈 사용자 메시지**: 사용자가 스페이스바만 입력하고 전송했을 때 이를 빈 메시지로 처리하는가.
189. **시스템 프롬프트 노출**: 특정 설정으로 인해 시스템 프롬프트가 `role: system`으로 내려올 때 이를 UI에 띄울지 판단.
190. **메타데이터 전용 유저 메시지**: 텍스트 없이 파일만 업로드한 사용자 메시지가 정상 처리되는가.

## [그룹 20: 레거시 및 미래 호환성 (Legacy & Future-proofing)] (191-200)
191. **구형 Model Slug**: `gpt-3.5-turbo`, `text-davinci` 등 옛날 모델 이름이 들어있어도 파서가 정상 작동하는가.
192. **미래 Model Slug**: `gpt-6`, `o2-pro` 등 파서가 모르는 새로운 모델 슬러그가 들어올 때의 대응.
193. **Deprecated 필드**: 현재는 쓰이지 않는 `end_turn`, `weight` 등의 레거시 필드를 무시하는가.
194. **새로운 Role**: 'assistant', 'user', 'tool' 외에 'agent', 'orchestrator' 등의 새로운 롤이 등장했을 때의 폴백(Fallback).
195. **새로운 Message Type**: `type: "image"` 등 텍스트가 아닌 새로운 기본 타입 메시지의 처리.
196. **스키마 버전업**: JSON 페이로드의 루트 구조가 배열에서 객체로 바뀌었을 때(또는 그 반대)의 파서 유연성.
197. **RSC 프로토콜 변경**: React 19의 새로운 RSC 직렬화 포맷을 파서가 감당할 수 있는가.
198. **플러그인 쇠퇴**: 더 이상 사용되지 않는 ChatGPT Plugin 포맷의 과거 대화 파싱 호환성.
199. **Canvas/Code Interpreter 통합**: 향후 Canvas 기능과 Code Interpreter가 합쳐진 새로운 메타데이터 구조 대비.
200. **에러 메시지 표준화**: 파싱 실패 시 발생하는 Error 객체가 향후 에러 트래킹 시스템에 넣기 좋게 규격화되어 있는가.
